K-Fusion Phase Timing Analysis
===============================
Date: 2026-03-09
Phase: 3A profiling baseline

Instrumentation
---------------
Added per-phase timing counters to col_op_k_fusion() in columnar_nanoarrow.c.
Four sub-phases measured via clock_gettime(CLOCK_MONOTONIC):

  alloc_ns    - calloc of results[], workers[], worker_sess[] arrays
  dispatch_ns - submit loop (wl_workqueue_submit x K) + wl_workqueue_wait_all barrier
  merge_ns    - col_rel_merge_k() + eval_stack_push()
  cleanup_ns  - result/mat_cache/arr/darr free loops + free() calls

Counters accumulated across all col_op_k_fusion calls per eval pass,
reset at evaluation start, exposed via col_session_get_perf_stats().

Analysis
--------

FINDING: Dispatch phase dominates both workloads.

  Andersen K=4: dispatch = 86.3% of kfusion time = 68.5% of total wall time
  CSPA K=2:     dispatch = 99.97% of kfusion time = ~99% of total wall time

The "dispatch" phase covers:
  1. wl_workqueue_submit x K (negligible - just enqueuing tasks)
  2. wl_workqueue_wait_all  (the actual parallel worker execution)

This means the dominant cost is the WORKER EXECUTION itself, not
synchronization overhead. The barrier wait (wl_workqueue_wait_all) is
merely waiting for K workers to finish their join/consolidate operations.

What this tells us:
  - alloc (calloc):  0.0-0.3%  → malloc pooling will NOT help
  - dispatch/barrier: ~86-100% → the workers ARE doing expensive work
  - merge (merge_k):  0.0-5.4% → merge algorithm optimization will NOT help
  - cleanup:          0.0-5.2% → cleanup optimization will NOT help

The 10-12% speedup target is lost inside the worker evaluation (join + consolidate),
not in the orchestration overhead. Specifically:

For CSPA: workers compute join(deltaR, S) and join(R, deltaS), then
consolidate results. With K=2 and a 200-fact dataset that expands to
20,381 tuples over 6 iterations, the join cost scales with relation size.

For Andersen: K=4 workers each evaluate one delta body of pointsTo.
The 68.5% wall-time spent in dispatch reflects genuine parallel join work,
not synchronization overhead.


Optimization Priority (based on data)
--------------------------------------

CORRECT PRIORITIES (informed by timing):
  1. Reduce worker join cost  — the dominant expense; options:
       a. Hash join (arrangement) already implemented (Phase 3C)
          Check if arrangements are being used in K-fusion worker sessions
       b. Larger delta batching: fewer but larger K-fusion calls
       c. Worker result sharing: avoid redundant sub-expression evaluation
          across the K workers (common sub-expression elimination)

  2. Merge (merge_k): 5.4% in Andersen — worth micro-optimizing only after
     join cost is addressed; SIMD merge or galloping merge could help here

INCORRECT PRIORITIES (data shows these are < 1%):
  - malloc pooling for alloc phase
  - Lock-free queue improvements for submit overhead
  - Cleanup/free optimization

KEY QUESTION FOR NEXT PHASE:
Are K-fusion workers using arrangement (hash) joins or sort-merge joins?
If the arrangement cache (arr_entries) is zeroed per-worker (it is, per the
code: arr_count/arr_cap = 0 in worker_sess setup), workers must rebuild
arrangements from scratch on every K-fusion call, even if the parent session
has arrangements cached. This may be the root cause of the 86-100% dispatch cost.

Verifying this hypothesis:
  - Check col_op_join path inside workers: does it hit arr_entries or fall back
    to sort-merge?
  - If fallback: sharing arrangements across workers (read-only) could
    eliminate the arrangement rebuild cost and recover the 10-12% speedup.
